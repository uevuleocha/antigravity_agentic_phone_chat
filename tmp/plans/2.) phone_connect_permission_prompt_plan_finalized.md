## Context

The Antigravity 2.0 agentic desktop app pauses agent execution and presents the user with a modal-style ask_question dialog when it needs a runtime decision. This blocks the agent until the user responds.

**Problem:** The current Phone Connect snapshot system only captures `[data-testid="conversation-view"]`. When a dialog is open, the phone user sees `"Waiting for user input..."` text in the snapshot (confirming the state is visible in the conversation flow), but the interactive options (numbered choices + Submit button) are rendered inside the same container. The agent is blocked and only the desktop can unblock it — the phone is read-only.

**Goal:** Detect when a runtime ask_question dialog is open, extract the question and options, push them to the phone as a dedicated overlay, and route the phone user's response back to the desktop via CDP to unblock the agent.

### Key Findings
- `data-tooltip-id` values are **dynamic** (`:r18v:`, `:r18u:`) — NOT usable for stable selection
- Submit button is uniquely identifiable by: `bg-accent` class + text `"Submit"` + being inside the dialog container
- Skip button: text `"Skip"` + `text-muted-foreground` class
- Numbered options: clickable items with text like `"1\nYes"`, `"2\nNo"`, `"3\nOther"`
- The write-in "Other" option reveals a text input after clicking — this is the custom response path
- Detection anchor: `"Waiting for user input..."` text in a `DIV` with class `select-none`

## Architecture

### Detection (server.js — inside `startPolling()`)
A secondary scan runs each polling cycle alongside the main snapshot. It queries `conversation-view` for the `"Waiting for user input..."` anchor element. When found, it extracts:
- The question/prompt text above the options
- All numbered option button texts
- Whether a "Skip" / write-in option exists

This data is attached to the `/snapshot` response as a new `permissionDialog` field — `null` when no dialog is active.

### Phone UI (app.js + index.html)
When `permissionDialog` is non-null, `app.js` renders a dedicated overlay modal (not inside the iframe) on top of the snapshot view. The overlay shows:
- The prompt/question text
- One button per option
- A text input + Submit button for the write-in path (shown when "Other"/"Skip" is selected)

### Response Routing (app.js → /answer-permission → server.js → CDP)
Tapping an option or submitting a write-in fires a `POST /answer-permission` with `{ optionIndex, customText? }`. The server:
1. Locates the correct numbered option element in the live DOM
2. Dispatches a click event on it
3. If write-in: types the custom text into the text input
4. Locates and clicks the Submit button (by `bg-accent` class + "Submit" text)

## Prompt Contract

```
GOAL:
  (1) Detect active ask_question / permission dialog in conversation-view via polling
  (2) Extract question text and options into structured JSON
  (3) Push dialog data to phone as permissionDialog field in /snapshot response
  (4) Phone shows full-screen overlay modal when permissionDialog is non-null
  (5) Phone user taps option → POST /answer-permission → CDP click on option element + Submit
  (6) Write-in path: user types custom text → injected into write-in input → Submit clicked
  (7) Dialog overlay auto-dismisses when permissionDialog returns null (dialog answered)

CONSTRAINTS:
  - Detection is polling-based (same 1s interval as snapshot) — NOT event-driven
  - Submit button targeted by class bg-accent + text "Submit" (not dynamic tooltip IDs)
  - Numbered options targeted by text pattern ("1\nYes", "2\nNo", etc.)
  - All server.js changes marked // [Permission Prompt Fix - PhaseX]
  - Phone overlay is a native HTML element in index.html — NOT inside the snapshot iframe
  - IDE compatibility preserved (if no dialog detected, existing snapshot flow unchanged)
  - No new npm dependencies
```
